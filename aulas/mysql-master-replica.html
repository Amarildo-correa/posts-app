MySQL Master/Replica — arquitetura de 10 passos

Guia de implementação da separação entre leitura e escrita no posts-app usando dois containers Docker independentes.

infraestruturaPasso 1 — configuração do MySQL
Você cria a pasta mysql/ com dois arquivos .cnf — um ensina o MySQL a ser Master, outro a ser Replica. Sem esses arquivos, os dois containers seriam MySQL genéricos sem identidade na replicação.
mysql/master.cnf→ ativa binary log, define server-id = 1, habilita replicação de escrita
mysql/replica.cnf→ ativa relay log, define server-id = 2, read_only = ON
O server-id é como o identificador único do servidor na topologia de replicação. Se dois tiverem o mesmo, o MySQL rejeita a conexão. O Master usa 1 por convenção universal — a Replica usa 2, 3, 4... para múltiplas réplicas.
configuraçãoPasso 2 — atualizar o .env
O .env agora tem dois hosts de banco em vez de um. Cada pool do Node.js vai apontar para um hostname diferente — Master para escrita, Replica para leitura.
DB_HOST_MASTER= mysql_master → recebe INSERT, UPDATE, DELETE
DB_HOST_REPLICA= mysql_replica → recebe SELECT do feed e perfil
REPLICATION_USER= usuário com privilégio exclusivo REPLICATION SLAVE
REPLICATION_PASSWORD= senha do usuário de replicação — vem do .env
O .env.example sobe pro GitHub com os nomes das variáveis — sem os valores reais. É o mapa das variáveis sem os segredos. Qualquer dev que clonar o projeto sabe quais variáveis precisa configurar.
scriptPasso 3 — setup-replication.js
Script Node.js em src/scripts/ que roda uma única vez para configurar a replicação. Cria o usuário de replicação no Master e aponta a Replica para o Master.
conecta no Master (root)
→
cria usuário replicador
→
lê posição do binary log
conecta na Replica
→
aponta pro Master (CHANGE REPLICATION SOURCE)
→
START REPLICA
É idempotente — se a Replica já estiver rodando com Slave_IO_Running = Yes, o script detecta e encerra sem fazer nada. Pode rodar toda vez que o container sobe sem causar problema.
orquestraçãoPasso 4 — docker-compose.yml BASE
O arquivo base agora orquestra três serviços em vez de dois. O Node.js só sobe depois que ambos os bancos passaram no healthcheck.
app_posts→ depends_on: mysql_master E mysql_replica (condition: service_healthy)
mysql_master→ volume mysql_master_data + master.cnf montado como :ro
mysql_replica→ volume mysql_replica_data + replica.cnf montado como :ro
Dois volumes completamente separados — os dados nunca compartilham disco. A sincronização acontece só via rede pelo protocolo de replicação nativa do MySQL (binary log → relay log). O :ro nos arquivos .cnf garante que o container não pode modificar sua própria configuração em runtime.
orquestraçãoPasso 5 — docker-compose.override.yml DEV
Em desenvolvimento, o command sobrescreve o CMD do Dockerfile e executa três etapas em sequência antes do servidor subir.
setup-replication.js
&&
migrate.js
&&
npx nodemon server.js
O && entre os comandos garante que cada etapa precisa ter sucesso (exit code 0) antes da próxima começar. Se a replicação falhar, o migrate não roda. Se o migrate falhar, o servidor não sobe. É a sequência de segurança que o Reddit usa antes de qualquer deploy — nunca sobe código em banco inconsistente.
orquestraçãoPasso 6 — docker-compose.prod.yml PRODUÇÃO
Em produção, sem command no Compose — o CMD do Dockerfile é quem executa a sequência. Sem volumes de código fonte — a imagem é autossuficiente e imutável.
NODE_ENV = production→ logs mínimos, performance máxima, sem erros detalhados expostos
sem volumes de código→ COPY . . no Dockerfile já empacotou tudo dentro da imagem
Princípio de imutabilidade — a imagem carrega tudo dentro. Qualquer container recriado da mesma imagem é idêntico ao anterior. É o que o Facebook chama de "infrastructure as immutable artifact" — você não edita o servidor, você substitui a imagem.
receitaPasso 7 — Dockerfile
O CMD do Dockerfile agora executa três etapas em sequência para produção. Em desenvolvimento, o command do override.yml substitui completamente esse CMD.
node src/scripts/setup-replication.js
&&
node src/scripts/migrate.js
&&
node server.js
O BuildKit cache mount --mount=type=cache,target=/root/.npm no RUN npm install persiste o cache do npm entre builds. Quando o package.json muda, o npm encontra os pacotes anteriores e instala só o que é novo — de ~40s para ~5s por deploy.
códigoPasso 8 — database.js
O coração da separação Master/Replica no Node.js. Dois pools — um aponta para o Master, outro para a Replica. O código de negócio só chama poolEscrita ou poolLeitura.
poolEscrita→ mysql_master | connectionLimit: 5 | 10% do tráfego (INSERT, UPDATE, DELETE)
poolLeitura→ mysql_replica | connectionLimit: 20 | 90% do tráfego (SELECT do feed e perfil)
O limite de conexões é assimétrico por design. Leitura tem mais conexões porque é 90% do tráfego. Mais conexões no Master = mais contenção de locks nas escritas = mais lentidão. O Reddit usa até 100 conexões por Replica em horário de pico — e apenas 5 a 10 no Master.
scriptPasso 9 — migrate.js
O migrate.js agora conecta explicitamente no Master via DB_HOST_MASTER. Migrations são operações de escrita — CREATE TABLE, INSERT, ALTER. A Replica recebe essas mudanças automaticamente via replicação.
host: process.env.DB_HOST_MASTER→ sempre o Master — nunca a Replica
Se conectasse na Replica, o read_only = ON bloquearia todos os CREATE TABLE e INSERT com erro ERROR 1290: The MySQL server is running with the --read-only option. Isso causaria falha imediata na inicialização — o servidor nunca subiria.
exclusãoPasso 10 — .dockerignore
A pasta mysql/ com os arquivos .cnf é configuração de infraestrutura — não pertence dentro da imagem Docker. O servidor Node.js não precisa saber como o MySQL foi configurado.
mysql/→ adicionado ao .dockerignore — não entra na imagem Node.js
Os arquivos .cnf são montados via volume diretamente no container MySQL pelo docker-compose.yml — não precisam estar dentro da imagem Node.js. Imagem menor = contexto de build menor = daemon recebe menos dados = build mais rápido.
