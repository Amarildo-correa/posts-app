Fluxo completo de inicialização

O que acontece por baixo desde o comando até o servidor Node.js conectado ao MySQL Master e Replica.

1
Docker lê o .env e o docker-compose.yml
.env (pasta raiz — fora do Docker — nunca entra na imagem)
DB_HOST_MASTER= mysql_master
DB_HOST_REPLICA= mysql_replica
DB_NAME= posts_app_db
DB_USER= posts_user
DB_PASSWORD= ••••••••
DB_ROOT_PASSWORD= ••••••••
REPLICATION_USER= replicador
SESSION_SECRET= ••••••••
docker-compose.yml
distribui variáveis para os containers
O Compose lê o .env e injeta cada variável no process.env de cada container via environment:. O Node.js, o MySQL Master e o MySQL Replica nunca tocam o arquivo .env diretamente — recebem tudo pronto via Docker.
2
Build das imagens
Dockerfile
imagem do Node.js
FROM node:20-alpine → base mínima Linux
COPY package.json → só o manifesto
RUN npm install → BuildKit cache mount (~5s com cache)
COPY . . → código fonte entra na imagem
.dockerignore bloqueia node_modules/, .env, mysql/
mysql:8.0
imagem do MySQL
Imagem oficial do Docker Hub — sem Dockerfile necessário. Versão fixada em 8.0 — nunca latest. Usada para criar os dois containers: mysql_master e mysql_replica. Mesma imagem, IPs internos diferentes.
3
Criação dos volumes
mysql_master_data
disco exclusivo do Master
Volume persistente montado em /var/lib/mysql do container mysql_master. Sobrevive ao docker compose down. Só some com docker compose down -v. Completamente isolado do volume da Replica.
mysql_replica_data
disco exclusivo da Replica
Volume persistente montado em /var/lib/mysql do container mysql_replica. Dados chegam via rede — nunca via disco compartilhado. O relay log vive aqui.
4
Containers MySQL sobem
mysql_master
IP interno único — porta 3306
Lê master.cnf montado como :ro. Ativa binary log (mysql-bin). Define server-id = 1. Cria banco posts_app_db automaticamente. Healthcheck verifica via mysqladmin ping a cada 5s.
mysql_replica
IP interno único — porta 3306
Lê replica.cnf montado como :ro. Ativa relay log (mysql-relay-bin). Define server-id = 2. Ativa read_only = ON. Sobe só depois que o Master passa no healthcheck.
mesma imagem mysql:8.0 — IPs diferentes na rede interna Docker — porta 3306 sem conflito
5
setup-replication.js — roda uma vez e encerra
Master (root)
cria usuário replicador
Conecta no Master como root. Cria usuário com privilégio exclusivo REPLICATION SLAVE. Lê posição atual do binary log.
Replica (root)
aponta pro Master
CHANGE REPLICATION SOURCE TO informa endereço, usuário e posição do binary log de onde começar a ler.
verificação
START REPLICA
Inicia IO Thread e SQL Thread. Verifica se ambos estão com status Running = Yes. Encerra com exit(0).
após esse passo — IO Thread e SQL Thread ficam vivos dentro do container mysql_replica continuamente
6
migrate.js — roda uma vez e encerra
Master (escrita)
cria migrations_executadas
Conecta no Master via DB_HOST_MASTER. Cria a tabela de controle se não existir. Consulta quais migrations já rodaram.
migrations/
executa só os arquivos novos
Lê pasta migrations/ em ordem numérica. Filtra local_* em produção. Executa só o que ainda não está na tabela de controle.
Replica (auto)
recebe via replicação
Todos os CREATE TABLE e INSERT do Master chegam automaticamente na Replica via binary log. Nenhuma configuração extra.
7
node server.js — fica vivo
poolEscrita
→ mysql_master : 3306
connectionLimit: 5 — 10% do tráfego. Recebe INSERT, UPDATE, DELETE. Stored Procedures de escrita. Nunca recebe SELECT do feed.
poolLeitura
→ mysql_replica : 3306
connectionLimit: 20 — 90% do tráfego. Recebe SELECT do feed, perfil, busca. Nunca recebe operações de escrita — read_only = ON bloquearia.

localhost:3000 disponível
