Fluxo completo de comunicação em uma aplicação web moderna — do clique do usuário ao banco de dados

Camada 1 — o usuário clica
Tudo começa no navegador. O usuário abre o feed do bairro no Comentaaê. O navegador precisa buscar esse HTML em algum servidor.
browserrequisição HTTP GET
O navegador monta uma requisição HTTP e envia para o servidor:

GET /feed/pinheiros HTTP/1.1
Host: comentaae.com
Cookie: sessionId=abc123

O cookie de sessão vai junto automaticamente — é assim que o servidor sabe quem é o usuário.
↓ viaja pela internet criptografada via HTTPS (Cloudflare SSL)
cloudflaretermina o SSL e repassa
O Cloudflare decifra o HTTPS e encaminha a requisição para o servidor Vultr via HTTP interno. Injeta o header X-Regiao: sudeste baseado no IP do usuário.
↓ chega no servidor Vultr
nginxroteia para o Node.js
O Nginx recebe na porta 80/443 e roteia internamente para o container app_posts:3000. O usuário nunca acessa o Node.js diretamente — o Nginx é a porta de entrada.
Camada 2 — o Node.js processa
A requisição chegou no Express. Agora ela percorre o pipeline MVC: middleware → route → controller → model → banco.
middlewareverificação de sessão
O auth.middleware.js intercepta a requisição antes do controller. Lê o cookie sessionId, consulta o Redis ou o banco para validar a sessão, e injeta o usuário autenticado no objeto req.usuario.

Se a sessão for inválida, retorna 401 Unauthorized antes de chegar no controller. O controller nunca vê requisições não autenticadas.
↓ sessão válida — passa para o controller
controllerorquestra a lógica
O post.controller.js recebe req e res. Ele não sabe nada de SQL — só chama o Model e decide o que fazer com o resultado:

const posts = await PostModel.buscarFeed(bairro)
res.render('feed', { posts, usuario })
↓ chama o model
modelconhece o banco
O post.model.js usa o poolLeitura — porque é uma operação de SELECT. Ele monta a query ou chama a Stored Procedure e retorna os dados pro controller. O model nunca sabe como os dados serão renderizados — só busca e retorna.
Camada 3 — o pool de conexões
Antes de falar com o banco, o Node.js precisa de uma conexão. O pool gerencia esse recurso de forma eficiente — sem abrir e fechar conexões a cada requisição.
sem poolconexão por requisição
1. abre conexão TCP com MySQL
2. autentica usuário e senha
3. executa a query
4. fecha a conexão

Custo de abertura: ~50ms
1.000 req/s = 1.000 conexões/s
MySQL colapsa
com poolconexões reutilizadas
1. pega conexão do pool (~0ms)
2. executa a query
3. devolve pro pool

20 conexões abertas permanentemente
1.000 req/s compartilham as 20
MySQL estável
database.jsdois pools separados por responsabilidade
poolEscrita → mysql_master — connectionLimit: 5
poolLeitura → mysql_replica — connectionLimit: 20

O TLS está ativo nas duas conexões — os dados trafegam criptografados dentro da rede Docker. Cada pool mantém as conexões abertas permanentemente e as distribui sob demanda.
Camada 4 — Master e Replica
O pool escolheu a conexão certa. Agora a query chega no banco. Escrita vai pro Master, leitura vai pra Replica — e os dois ficam sincronizados continuamente via replicação.
mysql_masterrecebe escrita
INSERT, UPDATE, DELETE

Executa no próprio banco
Registra no binary log:
posição 875: INSERT posts...

Confirma pro Node.js
Usuário vê: "post publicado"
mysql_replicarecebe leitura
SELECT — feed, perfil, busca

Nunca recebe escrita
read_only = ON

Serve 90% do tráfego
connectionLimit: 20
locks de escrita: zero
↓ replicação assíncrona contínua via TCP/IP
IO Thread + SQL Threadsincronização em pipeline
IO Thread: conectado ao Master via TCP — baixa eventos do binary log — salva no relay log
SQL Thread: lê o relay log — aplica os eventos nas tabelas da Replica

Os dois rodam em background com prioridade baixa — nunca competem com as queries de leitura dos usuários. Latência de sincronização: ~10ms.
Camada 5 — a resposta volta
A query executou, os dados voltaram pro Node.js. Agora o Express monta a resposta e envia de volta para o usuário.
SSR — resposta HTML completaHandlebars renderiza o template
O controller chama res.render('feed', { posts }). O Handlebars combina o template com os dados e gera HTML completo. O Node.js devolve:

HTTP/1.1 200 OK
Content-Type: text/html
<html>...feed completo...</html>

O Googlebot recebe esse HTML e indexa o conteúdo. O usuário vê o feed instantaneamente sem esperar JavaScript.
↓ depois do HTML chegar no browser
hidratação — Fetch APIinterações dinâmicas via SPA
O JavaScript hidrata a página. Curtidas, comentários e interações usam fetch('/api/curtir', ...) — sem recarregar o HTML inteiro. O Nginx roteia essas chamadas para a porta da API:

POST /api/curtir/123 → poolEscrita → mysql_master
GET /api/feed/cursor/875 → poolLeitura → mysql_replica
WebSocket — notificações em tempo realconexão persistente bidirecional
Quando alguém comenta no post do usuário, o servidor precisa avisar sem que o usuário precise recarregar. O WebSocket mantém uma conexão aberta permanentemente — o servidor empurra eventos pro browser sem que o browser precise pedir.

HTTP: browser pede → servidor responde → conexão fecha
WebSocket: conexão aberta → servidor envia quando quiser

Do clique do usuário ao dado no banco e de volta. Cada seta é um protocolo diferente, cada camada tem uma responsabilidade única.
